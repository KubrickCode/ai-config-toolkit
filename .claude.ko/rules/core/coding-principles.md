# 코딩 원칙

## 하나의 함수, 하나의 책임

- 함수명이 "and" 또는 "or"로 연결되면 분리 신호
- 각 if 분기에 테스트 케이스가 필요하면 분리 신호

## 조건문과 루프 깊이 2단계 제한

- 가능한 한 early return으로 깊이 최소화
- 그래도 무거우면 별도 함수로 추출

## 함수 부작용 명시화

- 예: `getUser`가 `updateLastAccess()`도 실행하면 함수명에 명시

## 매직 넘버/문자열은 상수로 변환

- 사용되는 파일이나 클래스 상단에 선언
- 많으면 상수 파일로 분리 고려

## 함수 순서는 호출 순서대로

- 언어별 명확한 컨벤션이 있으면 따르기
- 없으면 호출 순서대로 위에서 아래로 배치

## 복잡한 구현은 외부 라이브러리 검토

- 로직이 복잡하고 테스트가 비대해질 때
- 업계 표준 라이브러리가 존재할 때
- 보안, 정확성, 성능 최적화가 중요할 때
- 플랫폼 호환성이나 엣지 케이스가 많을 때

## 모듈화 (코드 중복 및 패턴 반복 방지)

- 코드 반복 절대 금지
- 유사한 패턴은 재사용 가능한 형태로 모듈화
- 재사용이 확정되면 선제적 모듈화 허용
- 과도한 추상화 피하기
- 모듈화 수준:
  - 같은 파일: 별도 함수로 추출
  - 여러 파일: 다른 파일/패키지로 분리
  - 여러 프로젝트/도메인: 패키지/모듈로 분리
